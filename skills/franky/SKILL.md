## Overview

This skill encapsulates the repeatable workflow for turning an existing HTML info card into a published `infocard-pub` page.

It also owns the broader release-engineering lane for Nick Fury's team: GitHub Actions, GitHub Pages, cron jobs, Docker/service deployment, CI/CD, logs/alerts, and post-release verification.

## DevOps / 发布工程师职责

- **GitHub Actions**：workflow 编写、触发、失败诊断、缓存与 artifact 检查。
- **Pages 发布**：GitHub Pages 发布、索引更新、CDN/缓存验证、404 诊断。
- **Cron job**：定时任务创建、更新、健康检查、失败告警。
- **Docker / 服务部署**：容器构建、启动、日志、端口、健康检查。
- **CI/CD**：测试、构建、发布、回滚、版本号与 changelog。
- **日志与告警**：发布后观察日志、错误率、可用性与告警信号。
- **发布后验证**：公网 URL、页面内容、索引入口、截图/DOM/CDP 验证。

It is designed for cases where the HTML already exists and the job is operational: place the file in the correct repository location, create or update the companion metadata, ensure the listing/index reflects the new page, push to Git, verify the public URL, and only then report the link back to the user.

The workflow is conservative by default: it prioritizes correctness, reproducibility, and link verification over speed.

## When to Use

Use this skill when the user asks to:
- publish an HTML info card to `infocard-pub`
- add the card to the public listing / index
- commit and push the page to GitHub
- verify that the public URL is reachable before responding
- generate or maintain the repository metadata alongside the HTML

Do not use this skill for:
- generating the card content itself
- redesigning the card layout from scratch
- investigative research without a publishing target
- unrelated GitHub workflows

## Default Operating Model

This skill assumes the following publishing contract:
- source input is an existing `.html` file
- the card must be added to the `infocard-pub` repository structure
- the matching `.meta.yaml` must be created or updated
- the repository listing must include the new card when required by the repo workflow
- changes must be committed and pushed
- the public GitHub Pages URL must be checked after deploy
- the final response must include the verified URL only after the page is reachable

## Publishing Workflow

### 1) Inspect the source HTML
Confirm the source HTML exists and identify:
- title
- slug / filename pattern
- category path
- date
- tags
- whether an accompanying screenshot or preview image should be preserved

### 1.5) Decide whether homepage indexing is required
If the repository uses a generated `_index.yaml` or a client-rendered index page, verify whether the new card is already included in the homepage listing. If not, update the index or wait for the index workflow, then verify the homepage actually shows the new entry before reporting success.

See `references/cdp-verification.md` for the safe browser verification recipe when the homepage target becomes stale or `Page.enable` times out.

### 2) Place files in the repository
Copy or write the HTML into the correct repo path.
Create the companion `.meta.yaml` with the same basename.

Typical layout:
- `docs/{YYYYMMDD}-{slug}.html`
- `docs/{YYYYMMDD}-{slug}.html.meta.yaml`

### 3) Update repository listings
If the repository uses an index/listing file or generated manifest, make sure the new entry is included according to repo conventions.

If the listing is generated by automation, do not hand-edit generated state unless the repository workflow explicitly requires it.

### 4) Commit and push
Use Git to:
- review the diff
- stage the new files
- commit with a clear message
- push to the active branch

### 5) Verify public availability
After push:
- open the GitHub Pages URL in a browser if available
- confirm the page loads successfully
- confirm the title / page content matches the published page
- if possible, verify the page path matches the repo slug exactly

### 6) Return the verified link
Only report the public URL after verification succeeds.
If the page is not reachable yet, wait or retry before responding with success.

## Common Pitfalls

1. **Forgetting the `.meta.yaml` file**
   - The repo workflow depends on a metadata sidecar; publishing only HTML may not surface the page correctly.

2. **Using the wrong repo path**
   - Confirm whether the active repo root is the canonical one used for Git operations, not a nested mirror or symlinked working copy.

3. **Assuming push = live**
   - Git push is not enough; always verify the GitHub Pages URL after deployment.

4. **Reporting a link before verification**
   - The final response should only include a link that has been checked in-browser or via a successful HTTP response.

5. **Skipping listing/index updates when required**
   - Some repo workflows require an index refresh or generated manifest update; follow the repository's current conventions.

6. **Creating HTML/meta but forgetting homepage indexing**
   - In this repo, the home index is not guaranteed to pick up new cards immediately unless `_index.yaml` is updated or the index workflow has completed. After publishing, verify the homepage actually shows the new entry before telling the user the card is publicly listed.
   - Run a quick check: `browser_navigate` the homepage URL and search for the card title in `document.body.innerText`. If not found, manually patch `_index.yaml` and push again. Never assume push = live in the index.
   - More critically: if the card HTML exists but the homepage still shows the old count, the `_index.yaml` is stale — do NOT rely on auto-rebuild if it hasn't fired yet. Manually add the new entry and push.

7. **Assuming the browser is still usable after a long session**
   - CDP targets can go stale (session dead, renderer crashed) while Chrome process remains alive. Signs: `Page.enable` or `browser_navigate` timeout repeatedly on the same target, even after retry.
   - Recovery: use `browser_cdp` to call `Target.createTarget(url="about:blank")` → get a fresh target ID → `Target.attachToTarget(flatten=True, targetId=...)` → then use `Page.enable` + `Page.navigate` on the fresh target. Do NOT keep retrying the same stale target.
   - For full Chrome restart, use `/home/ccwq/hehome/automation-chrome.sh`.
   - See `references/cdp-stale-target-recovery.md` under the `diagnose` skill for the complete recovery pattern.

8. **Assuming GitHub Pages is live immediately after push**
   - GitHub Pages can take **1-2 minutes** to rebuild after a new file is pushed. The `raw.githubusercontent.com` URL is available immediately (HTTP 200 within seconds); the `ccwq.github.io` Pages URL may return an empty page or 404 while Pages is rebuilding.
   - **Verification sequence**: try `raw.githubusercontent.com` first for instant confirmation → if that works, the file is committed correctly → then note "Pages may take 1-2 min to reflect the new file."
   - **Never** report the Pages URL as "live and verified" if browser shows "(empty page)" — the page is not ready yet.
   - Use `curl -sI` on both URLs to compare response timing if browser is stale.

9. **`_index.yaml` 被覆盖/截断 → 首页显示 0 张卡片（静默数据丢失）**
   - 根因：`Build Infocard Index` workflow 从 `*.meta.yaml` 重新生成 `_index.yaml`；但若手动执行了错误的 "rebuild index" commit（如只 push 了部分 meta.yaml，或 meta.yaml 缺少 tags），生成的 `_index.yaml` 会截断（196行→91行，16卡片→9卡片）。
   - 现象：公网 Pages 加载后显示"总卡片数 0 / 无匹配结果"，浏览器控制台无 JS 报错，`fetch('./_index.yaml')` 返回 200 但内容不足。
   - 诊断：`curl -s https://ccwq.github.io/infocard-pub/_index.yaml | grep "^- " | wc -l` → 应返回 16，少于此值说明截断。
   - 恢复：`git log --all --oneline _index.yaml` 找有完整内容的 commit（如 `c9740b0`），`git show <commit>:_index.yaml > _index.yaml`，检查所有 16 张卡片的 slug 都在，重新 push。
   - 验证：push 后等 Pages 重建，刷新浏览器或加 `?t=` 参数破坏缓存，确认"总卡片数"恢复为 16。

## Verification Checklist

- [ ] Source HTML identified and placed in the correct repo path
- [ ] Companion `.meta.yaml` created/updated
- [ ] Listing/index updated if required by repo workflow
- [ ] Git diff reviewed
- [ ] Commit created
- [ ] Push completed successfully
- [ ] Public URL verified reachable
- [ ] `_index.yaml` card count verified (`curl -s .../_index.yaml | grep "^- " | wc -l` ≥ expected count)
- [ ] Homepage shows correct card count (not 0)
- [ ] Final response includes the verified link

## Response Style

When communicating status to the user:
- be concise
- state what was published
- include both `raw.githubusercontent.com` URL (instant) and `ccwq.github.io` Pages URL (may need 1-2 min)
- mention any caveat only if it affects accessibility
- avoid speculative claims about deployment status
- attach the screenshot (MEDIA:/path/to.png) alongside the URL for immediate visual confirmation
